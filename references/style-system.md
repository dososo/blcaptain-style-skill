# 视觉系统

本文件是 BLCaptain Style Skill 的活跃视觉系统源头。后续 Agent、文档、示例和 gate 都必须以这里为准。

当前只允许两套主视觉系统：

1. `Still Paper`
2. `Signal Proof`

不得新增第三套主视觉系统。`BC-01 Bridge Canvas` 只是系统展示层，不能混入日常内容卡主题。`SL-05 Signal Noir` 仍是候选状态，状态为 `candidate-needs-visual-direction-board`，在补齐视觉方向稿、用户确认、方向 map 激活和 gate 通过前不得实现。

当前项目仍处在 `SP-C01 · Still Paper · SP-01 Mist Field` 硬闸门阶段。技术 PASS 不等于视觉 PASS；人工视觉确认没有通过前，不得进入 `SL-C02`、Core 4、Core 8 或开源发布。

## Still Paper

一句话定义：`Still Paper` 把内容变成克制、有纸感、有呼吸感的中文内容卡。

适合：

- 长文拆卡
- 公众号封面
- 小红书竖图
- 阅读笔记
- 创作者札记
- 旅行、生活方式、食物、桌面、家居
- 非虚构观察和带一点情绪的判断

视觉气质：

- 中文优先
- 纸感、纤维、微颗粒
- 低饱和、克制、安静
- 图片是情绪锚点，不是装饰
- 留白要有方向，不能空
- 字少但判断清楚

内容优先级：

1. 一个明确判断
2. 情绪和现场感
3. 摄影图作为叙事锚点
4. 简短导语或摘句
5. 页脚、来源、日期等轻量元信息

图片规则：

- 优先写实摄影、真实物件、自然光和有意境的环境图。
- 可以使用用户图、AI 生成的纯净摄影图、公开图源。
- 不得使用视觉方向稿裁切图作为生产 hero 图。
- 图片中不能带海报字、卡片边框、嵌入排版或不可控水印。
- 满铺图必须声明主体地图、文字安全区、避让区、遮罩和 `object-position`。
- 有人物时，文字不得压住人脸、手部动作或画面情绪核心。

常见失败：

- 变成无重点的浅色空卡
- 只有氛围没有信息
- 图片只是背景装饰
- 标题太散，用户不知道为什么停下
- 米色、灰绿、暖纸色堆在一起但没有层级
- 页脚和底注抢戏

## Signal Proof

一句话定义：`Signal Proof` 把工具、截图、数据和流程变成可信、可复用的证据卡。

适合：

- AI 工具介绍
- 截图教程
- 产品 walkthrough
- 工作流证明
- 数据复盘
- KPI、年度总结、增长分析
- 工具对比、清单、决策树
- 方法论和操作记录

视觉气质：

- 证据优先
- 结构清楚
- 截图可读
- 数据要有结论
- 网格和线条服务于判断
- 像可靠的产品系统，不像泛科技模板

内容优先级：

1. 证据或截图
2. 一个主判断
3. 清晰的信息结构
4. 来源、时间、上下文线索
5. 简短解释和最多两个重点 callout

图片规则：

- 截图必须够大，用户能看清关键 UI 或数据。
- 公开网页、产品截图、图表和工作流图必须记录来源。
- 不得把截图当模糊背景。
- 不得用假 dashboard、假日志、假代码制造可信感。
- 最多两个 callout；超过两个通常意味着布局或文案没有收敛。
- 数据组件必须有结论，不做无意义指标堆叠。

常见失败：

- 变成普通蓝白科技卡
- 模块太多且权重相同
- 截图太小不可读
- 表格密到像导出的报表
- 数据没有结论
- 线条、框、标签比内容更抢眼
- 看起来像临时 PPT 模板

## 两套系统的差异

| 维度 | Still Paper | Signal Proof |
|---|---|---|
| 核心任务 | 让内容值得停下阅读 | 让信息看起来可信可查 |
| 第一眼 | 克制、纸感、现场感 | 证据、结构、产品感 |
| 主资产 | 摄影、环境、物件、人物背影 | 截图、数据、表格、流程 |
| 文本密度 | 低到中 | 中等但高度结构化 |
| 情绪 | 安静、专注、回味 | 清晰、可信、可执行 |
| 主要平台 | 小红书、公众号封面、文章拆卡 | 小红书、公众号、教程卡、产品说明 |
| 最大风险 | 空、散、只有氛围 | 冷、满、像模板 |

## 共享原则

### 一张卡只做一件事

每张卡必须先回答它的任务：

- 吸引停留
- 证明一句话
- 解释一个方法
- 对比几个选择
- 总结一组数据
- 建立一种情绪

同一张卡同时做太多事，就要拆卡或换布局。

### 图片和文字必须协作

图片不是背景布。每张使用图片的卡都必须明确：

- 图片角色
- 裁切策略
- 主体区域
- 文字安全区
- 来源记录
- 是否需要在成图内显示来源

无图时一次性给用户 A/B/C 选择：

- A：用户自己提供图
- B：AI 生成纯净写实摄影图
- C：公开图源取图并记录来源

公开图源顺序固定为：

```text
Unsplash -> Pexels -> StockSnap CC0 -> Pixabay -> Kaboompics -> Flickr CC0 / Public Domain -> Openverse CC0 / Public Domain -> Wallhaven -> 直接搜索
```

### 文案必须压缩

默认社交卡不承载正文段落。推荐范围：

- 标题：8 到 18 个中文字符
- 导语：24 到 48 个中文字符
- 注释：1 到 2 行
- 模块文案：10 到 24 个中文字符
- 页脚：只放元信息

### 主题不是换色皮肤

主题会影响：

- 图片选择
- 纹理强度
- 字体对比
- 行距和密度
- 线条粗细
- 卡片表面
- 来源和底注的存在感

如果只换色、不改变图片策略和信息节奏，就是失败主题。

### 中英混排要有独立语言气质

中文是主体验收对象，但英文、数字、代码、UI label 不能完全依赖中文字体 fallback。

要求：

- 中文标题和正文使用中文友好的字重与字面。
- 英文关键词、数字、代码和 UI 文本应有独立的 Latin / mono 角色。
- Still Paper 的英文要温和、克制、像出版物旁注。
- Signal Proof 的英文要清晰、可靠、像产品系统标签。
- 字体环境未命中目标栈时，只能算工程预览，不能做最终视觉验收。

### 不混用系统

一个成图任务只能选择一个主系统。只有做产品总览、README hero 或能力地图时，才允许使用 `BC-01 Bridge Canvas` 做系统展示层。

### Skill 是结构化工作流

Skill 本身不是单纯生图器，而是固定流程：

1. Intake
2. Style & Theme
3. Layout Contract Selection
4. Asset Prep
5. Compose & Render
6. Gate & Review
7. Iterate

每一步都要有搜索、学习、研究、验证、测试、审计、洞察、分析。一个对象未通过，不进入下一对象。

## 命名规则

公开文档和用户可见输出只使用以下产品语言：

- `Still Paper`
- `Signal Proof`
- `BC-01 Bridge Canvas`

可以研究优秀开源项目和业界最佳实践的流程、方法、质量标准和产品化经验，但不得复制其设计、代码、模板、配色、资产或布局，也不得把我们的系统描述成别人的风格复刻。
